0. 공통 사항 (Common)

0.1 사전 조건 (Common Preconditions)

  • 테스트 계정이 KMS 서비스에 정상적으로 등록되어 있음
  • 테스트용 프로젝트/리전이 사전 세팅되어 있음
  • 충분한 권한 (KMS Admin, KMS User, Viewer)을 가진 계정이 각각 준비됨
  • 네트워크/세션 상태 정상

0.2 공통 UI/UX 검증 포인트

  • [ ] 페이지 로드 시 1차 콘텐츠가 2초 이내 표시되는지
  • [ ] 라이트 / 다크 테마 토글 시 모든 요소가 누락 없이 전환되는지
  • [ ] 한국어 UI에 잘림(truncation) 또는 줄바꿈 깨짐 현상이 없는지
  • [ ] 키보드 포커스 이동(Tab) 시 시각적 포커스 링이 명확히 보이는지
  • [ ] 모든 버튼/링크에 hover, active, disabled 상태가 정의되어 있는지
  • [ ] 오류·성공 토스트 메시지가 일관된 위치(우상단 또는 상단)에 노출되는지

0.3 용어

용어 정의
고객 관리형 키 (CMK) 고객이 직접 생성·관리하는 암호화 키
플랫폼 관리형 키 (PMK) 플랫폼이 자동으로 발급·운용하는 키
키 회전 (Rotation) 키의 암호 자료를 새로운 버전으로 갱신하는 작업
활성화 / 비활성화 키 사용 가능 여부 토글
서비스 해지 키와 연결된 KMS 서비스 자체 종료 (즉시 / 예약 / 취소)

1. Service Home

TC-SH-001 — 서비스 홈 진입 (정상)

항목 내용
사전 조건 KMS 사용 권한 보유 계정으로 로그인 완료
단계 1. 콘솔 좌측 네비게이션에서 Key Management Service 진입 → 2. Service Home 화면이 표시되는지 확인
기대 결과 • 헤더에 "Key Management Service" 타이틀 표시
• 좌측 GNB가 강조(active) 상태
• 메인 영역에 서비스 개요 카드/대시보드 정상 노출
UI/UX 검증 라이트/다크 테마 모두 색상 대비 4.5:1 이상

TC-SH-002 — 서비스 홈 주요 위젯 표시

항목 내용
사전 조건 TC-SH-001 완료
단계 1. 홈 화면의 위젯 영역 확인 → 2. 위젯별 제목·수치·CTA 버튼 표시 확인
기대 결과 • "최근 키", "이번 달 키 사용량", "공지/업데이트" 등 위젯 영역이 일관된 카드 레이아웃으로 표시
• 각 카드의 "더 보기" 액션이 해당 상세 페이지로 이동
UI/UX 검증 카드 간 여백 24px, 그림자/테두리 디자인 토큰 일치

TC-SH-003 — 서비스 홈에서 "키 생성" 진입 (CTA)

항목 내용
사전 조건 KMS Admin 권한 보유
단계 1. 홈 화면 상단 우측의 키 생성 CTA 클릭
기대 결과 • "고객 관리형 키" 생성 페이지(또는 키 종류 선택 모달)로 이동
• 새 탭이 아닌 동일 탭에서 라우팅
UI/UX 검증 CTA가 primary 색상, 아이콘 + 텍스트 정렬 정확

TC-SH-004 — Service Home 권한별 노출 차이

항목 내용
사전 조건 Viewer 권한 계정
단계 1. Viewer 계정으로 로그인 → 2. KMS Service Home 진입
기대 결과 • "키 생성" CTA 비활성화 또는 비노출
• Admin 전용 위젯이 가려지거나 안내 메시지 노출
UI/UX 검증 비활성 버튼은 tooltip으로 사유 표시

TC-SH-005 — 빈 상태(Empty State)

항목 내용
사전 조건 키가 한 건도 등록되지 않은 신규 프로젝트
단계 1. KMS Service Home 진입
기대 결과 • "아직 생성된 키가 없습니다" 일러스트 + 안내 문구 노출
• "키 생성" CTA가 강조 표시

TC-SH-006 — 다크 테마 전환

항목 내용
사전 조건 TC-SH-001 완료

기대 결과 • 배경, 카드, 텍스트, 차트 색상 모두 다크 토큰으로 전환
• 아이콘/이미지가 다크 환경에서도 가독성 유지

2. 고객 관리형 키 (Customer Managed Key)

2.1 키 리스트 조회

TC-CMK-LIST-001 — 키 리스트 정상 조회

항목 내용
사전 조건 1건 이상의 고객 관리형 키 등록되어 있음

기대 결과 • 테이블 컬럼: 키 이름, 상태(활성/비활성), 키 ID, 생성일, 자동 회전, 액션
• 페이지네이션 또는 무한 스크롤 정상 동작
• 기본 정렬: 생성일 내림차순
UI/UX 검증 헤더 sticky 동작, hover 행 강조

TC-CMK-LIST-002 — 검색 및 필터

항목 내용
사전 조건 10건 이상의 키 존재
단계 1. 상단 검색창에 키 이름 일부 입력 → 2. 상태 필터(활성/비활성) 선택 → 3. 자동 회전 필터 적용
기대 결과 • 입력 즉시 또는 Enter 시 결과가 필터링됨
• 필터 칩이 검색바 하단에 표시되고 X로 개별 해제 가능

항목 내용

단계 1. 컬럼 헤더(키 이름 / 생성일) 클릭
기대 결과 • 오름/내림차순 토글
• 정렬 표시(▲▼) 아이콘 노출

TC-CMK-LIST-004 — 페이지네이션

기대 결과 • 생성 성공 토스트 노출

TC-CMK-PRI-001 — 프라이빗 접근 허용 리소스 추가

2.8 서비스 해지

TC-CMK-TERM-001 — 즉시 해지

대응 디자인: 서비스 해지/즉시 해지

단계 1. 상세 → 우측 액션 서비스 해지 → 2. "즉시 해지" 선택 → 3. 키 이름 재입력 → 4. 해지
기대 결과 • 모든 사전 안내 항목 체크박스 체크 시에만 진행
• 해지 후 키 상태가 "해지됨"으로 변경, 모든 액션 비활성화
• 활동 로그 기록

TC-CMK-TERM-002 — 예약 해지

대응 디자인: 서비스 해지/예약 해지

단계 1. "예약 해지" 선택 → 2. 해지 예정일 캘린더로 선택 (예: 7일 뒤) → 3. 예약
기대 결과 • 상세 페이지 상단에 "YYYY-MM-DD 해지 예약됨" 배너 노출
• 예약일 도래 전까지 정상 동작
• 과거 날짜 선택 시 불가

TC-CMK-TERM-003 — 예약 해지 취소

대응 디자인: 서비스 해지/해지 취소

단계 1. 예약 해지 상태에서 배너의 해지 취소 클릭 → 2. 확인 모달
기대 결과 배너 사라지고 정상 상태 복귀, 토스트 표시

TC-CMK-TERM-004 — 해지 가능 권한 검증

단계 1. Viewer/일반 사용자 계정으로 해지 시도
기대 결과 서비스 해지 버튼이 비활성 또는 노출되지 않음

3. 플랫폼 관리형 키 (Platform Managed Key)

3.1 키 리스트 조회

TC-PMK-LIST-001 — 플랫폼 관리형 키 리스트

단계 1. GNB → 플랫폼 관리형 키 진입
기대 결과 • 컬럼: 키 이름, 서비스 매핑, 상태, 마지막 회전, 생성일
• 고객 관리형 키 리스트와 시각적으로 구분되는 헤더/뱃지

TC-PMK-LIST-002 — 검색·필터

단계 1. 서비스명/상태로 필터링
기대 결과 즉시 결과 갱신, 칩 표시

3.2 키 상세 조회

TC-PMK-DETAIL-001 — 상세 페이지 표시

단계 1. 리스트에서 키 클릭
기대 결과 • 메타데이터(키 ID, 서비스 매핑, 알고리즘) 표시
• "이 키는 플랫폼에서 자동으로 관리됩니다" 안내 배너
• 일부 액션(생성·삭제·수동 회전 등)은 비활성 또는 비노출

TC-PMK-DETAIL-002 — 활동 로그 / 회전 이력

단계 1. 상세 → 활동 로그 또는 회전 이력 탭
기대 결과 • 시간순(역순) 정렬
• 회전 이벤트는 "auto-rotation" 라벨로 구분

3.3 권한 / 노출

TC-PMK-PERM-001 — Viewer 권한

단계 1. Viewer 계정으로 진입
기대 결과 • 리스트·상세는 조회 가능
• 다운로드/내보내기 등 액션은 비활성화

TC-PMK-PERM-002 — 직접 조작 시도 (Negative)

단계 1. 플랫폼 관리형 키 상세에서 URL을 통한 강제 액션 시도 (예: 삭제 API 호출 흉내)
기대 결과 403 또는 "이 키는 플랫폼이 관리합니다" 에러 처리, UI 상에서도 진입 자체 차단

TC-PMK-I18N-001 — 한글·영문 라벨 일관성

4. 교차 시나리오 (Cross-feature)

TC-X-001 — 키 종류 간 이동 (네비게이션)

기대 결과 • 이동 후 이전 페이지 상태(필터/스크롤) 합리적 보존

TC-X-002 — 활동 로그 일관성

단계 1. CMK·PMK에서 발생한 이벤트가 통합 활동 로그(또는 각 상세) 에서 모두 조회 가능한지 확인
기대 결과 • 이벤트 ID / 행위자 / 시각 / 액션 / 대상 표기 일관
• 필터·검색 가능

TC-X-003 — 동시 편집 충돌 (Optimistic Concurrency)

단계 1. A/B 두 세션에서 동일 키 설명을 동시에 수정 → 2. 둘 다 저장
기대 결과 나중 저장 시 "이미 변경되었습니다. 새로 불러와 주세요" 안내

TC-X-004 — 세션 만료 처리

단계 1. 장시간 idle 후 키 수정 시도
기대 결과 만료 안내 모달 → 재인증 후 작업 복귀

5. UI/UX 시각 검증 체크리스트

5.1 레이아웃

  • [ ] 모든 페이지 사이드바·헤더 고정 동작 일관
  • [ ] 컨텐츠 영역 최소/최대 폭(1280–1600px) 가독성
  • [ ] 카드·테이블의 패딩·여백이 디자인 토큰과 일치

5.2 색상·타이포

  • [ ] Primary / Danger / Success / Warning 컬러 사용처 일관
  • [ ] 본문 14px / 보조 12px / 제목 18~24px 위계
  • [ ] 다크 모드의 배경 대비(WCAG AA) 만족

5.3 상태 표시

  • [ ] 활성 / 비활성 / 해지됨 / 해지 예약 / 삭제 진행 중 배지 색상·아이콘 구분
  • [ ] 로딩 스켈레톤이 ≥300ms 작업 시 노출
  • [ ] 빈 상태 / 에러 상태에 일관된 일러스트 사용

5.4 인터랙션

  • [ ] 모든 모달은 ESC 키 + 외부 영역 클릭으로 닫힘 (단, 위험 액션 모달은 외부 클릭으로 닫히지 않음)
  • [ ] 토스트는 4초 후 자동 사라짐 + 수동 닫기 가능
  • [ ] 폼 제출 중에는 버튼이 로딩 상태(spinner)로 전환되고 중복 클릭 방지

5.5 접근성

  • [ ] 모든 인터랙티브 요소 Tab 순서 자연스러움
  • [ ] 스크린리더용 aria-label 설정
  • [ ] 색상만으로 상태를 전달하지 않음 (텍스트/아이콘 병행)

7. 미정/추가 확인 필요 사항 (Open Questions)

Figma MCP 호출 한도 도달로 모든 프레임을 세부 분석하지는 못했습니다. 아래 항목은 디자인 원본을 추가 확인하여 시나리오를 보강할 필요가 있습니다.

  1. Service Home의 위젯 구성 — 정확한 카드 종류, 데이터 항목, CTA 명칭
  2. 고객 관리형 키 — "버전" 탭 — 키 버전(회전 이력)의 UI 구성과 가능한 액션 (수동 회전 여부 등)
  3. 태그/라벨링 — 키 생성 시 태그 입력 규칙(개수 제한, 키-값 형식 등)
  4. 다국어 범위 — 영어 외 추가 언어 지원 여부
  5. 알림(이메일/팝업) — 키 해지 예약·자동 회전 알림이 어떤 채널로 발송되는지
  6. 감사 로그 보존 기간 — 활동 로그가 몇 일/몇 개월 보존되는지

위 항목들이 확정되면 본 문서에 추가 케이스를 보강할 수 있습니다.

문서 끝. 변경 이력은 별도 관리.
